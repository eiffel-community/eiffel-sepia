Untangling the network of pipelines

Modern software production systems are more than just a build server. It is a shifting mix of functions and technologies, frmo issue trackers and test frameworks to environment managers and repositories. At the same time, software supply chains become increasingly complex, crossing organizational and geographical borders many times before resulting in the final software product. It is a world of interconnected and interdependent software pipelines.

The Eiffel community applies the same architectural principles to this network of pipelines as to any other software design problem. The first part of this is the Eiffel protocol, enabling technology agnostic event-based communication among the actors of the system. The second part is the implementation of services and plugins for sending, storing, analyzing and acting upon those event communications.

The Eiffel protocol and its implementaions lets you know at a glance what is up in your software production system and your supply chain, while providing rich extension points where others can hook into your pipeline with perfect upstream and downstream traceability.

Nimble yet designed for heavy duty

The Eiffel protocol saw the light of day in 2012 as a protocol for large-scale software integration within Ericsson. In 2016 Ericsson made the initial contribution of the protocol to open source, and has since then been collaborating with the community to create open source implementations of it. Coming from the world of large telecommunications systems it was designed for scalability and high demands on traceability. At the same time, the design of the protocol and its implementations ensures that you can cherry-pick the parts that are relevant to you. This affords an easy-to-adopt yet highly flexible and scalable solution that will grow along with your needs.

Getting started

To get into the concepts of the Eiffel protocol, subscribe to our Youtube channel and watch the screen casts on what the Eiffel protocol is, and how it achieves its mission. If you're looking for examples of how to use it, try the Usage Examples provided in the protocol repository. If you want to try it out yourself in a sandbox environment, try building and running the sandbox Docker image (to be added soon).

Keep in mind that Eiffel is designed to let you cherry-pick the parts that add value for you, while ignoring the rest. This goes for the protocol as well as for its implementations. Browse the protocol vocabulary and the Sepia implementation architecture to find what supports your critical use case, and then expand from there.

Getting involved

Contributions are more than welcome in the Eiffel community, both to the protocol itself and to its implementations. A good starting point for contributing to the protocol itself is its list of open issues: find an unaddressed one, or create a new one! To get involved on the implementation side, start by visiting the Sepia implementation architecture to find implementations you think are missing, or existing implementations you can contribute to. Every repository in the Eiffel community maintains an issue list, accepts pull requests and features a maintainers' mailing list (linked from each respository README).

The fundamentals

The Eiffel protocol is the foundation of everything else within the Eiffel community. It affords actors within the software production system an agreed and unambiguous vocabulary and syntax to communicate with one another.
Repository: eiffel
Status: Mature.

The Sepia Eiffel Protocol Implementation Architecture is one level down from the Eiffel protocol in terms of abstraction. Because the protocol makes no assumptions with regards to implementation and technology choices, creating a software production system utilizing the Eiffel protocol is a design problem with infinite solutions. Sepia narrows that solution space by identifying types of actors, reference implementations, interface specifications as well as data and Eiffel event type ownership. To get an overview of how actors may collaborate in practice using Eiffel and the concrete implementations available, Sepia is a good place to start.
Repository: eiffel-sepia
Status: In progress.

The meat

Eiffel events are small and atomic, but non-trivial use cases require more information than what can be gained from a single one. Eiffel Intelligence aggregates events over time into rich data objects, representing everything that is known to date about a given type of entity: release candidates, test cases, source revisions, developers and so on. Through its subscription interface clients can sign up for alerts whenever a certain state has been achieved, providing a powerful yet easy to use mechanism for acting upon data gathered from the software production system.
Repository: eiffel-intelligence
Role in Sepia: TBD
Status: Functional, but not mature.

Remrem acts as a REST based mailbox towards an Eiffel message bus. It provides an abstraction that allows its clients to send messages without first integrating with a particular messaging technology, and takes care of most of the heavy lifting involved in authoring and publishing event messages.
Repository: eiffel-remrem
Role in Sepia: TBD
Status: In progress.

Vici dynamically visualizes Eiffel retrieved from a persistance service. This affords overview and inspection of the software production system in real time, such as it really is, across underlying technologies. Repository: eiffel-vici
Role in Sepia: TBD
Status: Functional, but not mature.

The workshop

Apart from the projects listed above, there are many ongoing experiments and studies ongoing. These are projects that welcome contributions and that may well turn into proper implementations over time, but currently leave no guarantees as to maintenance or longevity.

The Eiffel JIRA Plugin turns JIRA into an Eiffel enabled actor in the software production system, able to emit events documenting the lifecycle of its tracked issues.

The icing

Outside of the Eiffel Community organization there are more implementations and extensions of the Eiffel protocol.

Praqma/tracey-bitbucket-eiffel-rabbitmq-post-receive-hook is a post receive hook plugin to BitBucket to send Eiffel messages using a RabbitMQ message broker.
